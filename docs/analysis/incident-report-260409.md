# Incident Report & Prevention Guide (2026-04-09)

오늘 발생한 주요 기술 이슈와 원인을 분석하고, 향후 재발 방지를 위한 엔지니어링 가이드를 정리합니다.

---

## 🚩 발생 이슈 요약 및 분석

### 1. AI 모델 식별자 불일치 (404 Not Found)
- **증상**: `gemini-1.5-flash` 또는 `gemini-1.5-flash-latest` 호출 시 404 에러 발생.
- **원인**: 구글 AI SDK 및 API v1beta 환경에 따라 에일리어스(Alias) 인식 규칙이 상이함.
- **해결**: 이력상 가장 안정적이었던 **`gemini-flash-latest`**로 모든 엔드포인트 통일.

### 2. 서버 과부하 대응 및 에러 메시지 개선 (503 Service Unavailable)
- **증상**: 무료 티어 사용량 급증 시 간헐적 503 오류 발생.
- **해결**: 
  - **재시도 루프**: 최대 2회 자동 재시도 로직 적용.
  - **UX 개선**: "서버 과부하로 나중에 다시 시도해 주세요"라는 친절한 한국어 문구로 대체.

### 3. PDF 파싱 라이브러리 연쇄 트러블
- **이슈 A (Worker)**: v2.x 버전이 Next.js 서버 환경에서 워커 모듈 로드 실패.
- **이슈 B (ENOENT)**: v1.1.1 버전이 로드될 때 내부 테스트 코드(`index.js`)가 실행되며 존재하지 않는 파일을 찾으려 시도.
- **해결**: `pdf-parse/lib/pdf-parse.js`를 직접 임포트하여 불필요한 초기화 로직을 우회하고 커스텀 타입(`.d.ts`) 정의.

## 🛡️ 향후 예방 및 대응 방안 (Prevention Strategy)

### [규칙] AI 모델 및 프롬프트 관리
- **검증된 식별자 사용**: 새로운 모델 도입 시 `scratch/check_models.ts`와 같은 진단 스크립트로 지원 여부를 먼저 확인한 후 적용할 것.
- **프롬프트 상수화 (GR-8)**: 모든 프롬프트는 반드시 `src/server/prompts.ts`에서 관리하여 API 로직과의 의존성을 분리할 것.

### [규칙] 외부 라이브러리 도입 가이드
- **Next.js 호환성 검증**: 오래된 CJS 라이브러리(예: `pdf-parse`) 도입 시, `index.js`의 부수 효과(Side Effects)를 반드시 확인하고 필요시 내부 로직만 선별적으로 로드할 것.
- **타입 정의 선제적 대응**: 내부 파일을 직접 참조할 경우 `src/types/`에 즉시 타입 선언을 추가하여 린트 에러를 방지할 것.

### [규칙] 방어적 프로그래밍 (Error Handling)
- **외부 API 호출**: 모든 외부 API 호출(AI, 스크래핑 등)은 '일시적 장애'를 상정하고 **재시도 루프**와 **사용자 친화적 폴백 메시지**를 기본적으로 포함할 것.
