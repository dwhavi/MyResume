# Golden Rules — MyResume

> 에이전트가 코드를 생성·수정할 때 **항상** 지켜야 할 불변의 원칙.
> 이 규칙은 취향이 아닌 아키텍처 안정성을 위한 것입니다.

## GR-1: 경계에서 파싱, 내부에서 타입 보장

데이터가 시스템 경계(API 요청, OpenAI 응답, localStorage 읽기)를 넘을 때 **반드시 Zod로 파싱**하세요. 내부에서는 파싱된 타입만 사용합니다.

```typescript
// ✅ 경계에서 파싱
const profile = ProfileSchema.parse(rawInput);

// ❌ 내부에서 raw 데이터 사용
function processProfile(data: any) { ... }
```

## GR-2: API 키는 절대 클라이언트에 노출하지 않음

`NEXT_PUBLIC_` 접두사를 사용한 환경변수에 API 키를 넣지 마세요. 모든 OpenAI 호출은 서버사이드 API Route에서만 수행합니다.

## GR-3: 한국어 이력서 항목 순서는 하드코딩

`기본정보 → 자기소개 → 학력 → 경력 → 기술스택 → 프로젝트 → 자격증`

이 순서는 프롬프트, UI 렌더링, PDF 레이아웃 모두에서 동일해야 합니다. 순서를 재배열하는 기능은 만들지 마세요.

## GR-4: AI가 사용자 정보를 만들어내지 않음

AI 응답에 사용자가 제공하지 않은 경력, 기술, 학력이 포함되어 있다면 버리세요. few-shot 예시나 프롬프트로 방어해도 완벽하지 않으므로, **서버에서 응답을 검증**할 때 프로필에 없는 항목이 생성 결과에 들어있는지 확인하는 검증 로직을 추가하세요.

## GR-5: 실패 시 빈 화면 금지

AI 생성 실패, 네트워크 오류, JSON 파싱 실패 — 어떤 경우든 사용자에게 빈 화면을 보여주지 마세요. 최소한 사용자가 입력한 프로필 기반의 기본 초안이나, 명확한 에러 메시지 + 재시도 버튼을 표시하세요.

## GR-6: 의존성 방향은 단방향

```
Types ← Schemas ← Lib ← API Routes ← Components ← Pages
```

- 위쪽 레이어는 아래쪽만 import 가능
- Components에서 `server/` 직접 import 금지
- API Routes에서 Components import 금지
- 순환 의존 절대 금지

## GR-7: 공유 유틸을 먼저 확인

새로운 유틸 함수를 만들기 전에 `lib/`에 이미 있는지 확인하세요. 중복 유틸을 만들면 일관성이 깨집니다. `lib/storage.ts`, `lib/api-client.ts` 등 이미 있는 것을 재사용하세요.

## GR-8: 프롬프트는 코드가 아닌 상수

프롬프트 문자열을 컴포넌트나 API Route 안에 인라인으로 작성하지 마세요. 모든 프롬프트는 `server/prompts.ts`에 상수로 분리하고, 버전 주석을 남기세요.

## GR-9: 파일 크기 제한

- 컴포넌트: 300줄 이상 → 분리 검토
- API Route: 150줄 이상 → 비즈니스 로직을 `server/`로 분리
- 프롬프트 상수 파일: 200줄 이상 → few-shot을 별도 파일로

## GR-10: 환경에 구애받지 않는 한글 처리

html2canvas로 PDF를 생성할 때, 한글 폰트는 시스템 기본에 의존합니다. 폰트 로딩을 기다리지 않으면 깨질 수 있으므로 `document.fonts.ready`를 항상 대기하세요.

## GR-11: 안정적인 AI 호출 및 에러 핸들링

모든 AI 관련 API Route는 다음을 의무적으로 포함해야 합니다:
- **자동 재시도**: 503(과부하) 등 일시적 장애 발생 시 지수 백오프 기반의 재시도 로직.
- **친절한 폴백**: 기술적인 에러 로그 대신 "서버 과부하로 나중에 다시 시도해 주세요"와 같은 사용자 친화적 메시지 출력.
- **모델명 통일**: 검증된 식별자인 `gemini-flash-latest`를 우선 사용.

## GR-12: 외부 라이브러리 도입 및 참조 제약

- **Side Effect 주의**: 오래된 CJS 라이브러리(`pdf-parse` 등) 도입 시, 초기화 과정에서 서버를 멈추게 하는 부수 효과가 있는지 확인하고 필요시 핵심 파일만 직접 참조할 것.
- **타입 가드**: 내부 파일을 직접 참조하거나 가변 데이터가 들어올 경우, 반드시 `src/types/`에 선언을 추가하거나 Zod로 검증할 것.
- **프로토타입 검증**: 새로운 기능을 추가하기 전, `scratch/`에서 독립적인 테스트를 거친 후 프로젝트 코드에 병합할 것.

## GR-13: 순차적 사고 및 단계적 실행

에이전트는 모든 복잡한 작업을 수행할 때 반드시 다음 단계를 준수해야 합니다:
1. **순차적 분석**: 변경할 범위를 먼저 분석하고 의존 관계를 확인합니다.
2. **단계적 계획**: 전체 작업을 작은 단위의 태스크로 나누어 논리적 순서에 따라 배치합니다.
3. **증분적 실행**: 한 번에 너무 많은 파일을 수정하지 않고, 각 단계가 완료될 때마다 상태를 확인하며 점진적으로 진행합니다.
4. **회고 및 보정**: 각 단계가 완료된 후 예상치 못한 부작용이 발생했는지 확인하고 필요시 계획을 즉시 수정합니다.
